### Explanation of Software Engineering Decisions and Security Compliance

1.  **Prepared Statements (R2, R5 Compliance):**
    *   The core of the solution relies on the `conn.fetch(base_query, *params)` method from `asyncpg`.
    *   User-supplied values (like `filter_params.name`, `min_price`, etc.) are never concatenated directly into the SQL string. Instead, they are passed as separate arguments (`$1`, `$2`, etc.) to the `fetch` function.
    *   **Result:** This forces the database driver (`asyncpg`) to handle the proper quoting, escaping, and type conversion for every parameter, completely neutralizing SQL injection attacks.

2.  **Dynamic WHERE Clause Construction:**
    *   The code dynamically builds the `WHERE` clause by checking which optional parameters are present (`if filter_params.name:`).
    *   We use a list (`where_clauses`) to collect individual conditions, which are then joined by `AND`. This pattern is robust for handling an arbitrary subset of filters.

3.  **Safe Sorting Implementation (R1 Compliance):**
    *   **Whitelisting:** The `ALLOWED_SORT_COLUMNS` list strictly defines which columns the user is permitted to sort by. If the user inputs a forbidden column, a `400 Bad Request` error is raised immediately, preventing the injection of malicious sorting logic.
    *   **Boolean Logic:** The `sort_order` input ("asc" or "desc") is explicitly mapped to the SQL keywords (`ASC` or `DESC`). This ensures the sorting logic is controlled by known, safe values, not arbitrary user strings.

4.  **Input Validation (Pydantic):**
    *   The `ProductFilter` Pydantic model enforces type safety (e.g., `min_price` must be a float, `sort_order` must be one of the allowed values if we added stricter validation, although here we rely on the explicit check in the endpoint). This ensures the data received by the function is already in a usable, predictable format.

5.  **Error Handling:**
    *   Specific HTTP exceptions (`400 Bad Request` for invalid sort columns, `500 Internal Server Error` for database issues) are raised, providing clean feedback to the client without leaking internal database errors.

     * that the resulting absolute path still starts with the BASE_DIR.
